## 🎯 Core Objectives

This project is designed to **emulate real physical routers and routing protocols** within a containerized environment, establishing a comprehensive and robust network topology.

| Objective | Network Engineering Approach |
|-----------|------------------------------|
| **Dynamic Routing & Redundancy** | Deployed **FRRouting** to run OSPF allowing native autonomous route discovery. |
| **Gateway High-Availability** | Implemented true **VRRP** (Virtual Router Redundancy Protocol). If Router 1 dies, Router 3 takes over exactly like physical Cisco/Juniper hardware. |
| **Branch Secure Tunneling** | Encrypted Site-to-Site payload using bare-metal **StrongSwan IPsec (IKEv2)**, protecting internal Subnets from simulated Internet ISPs. |
| **Microservice Decoupling** | Isolated DMZs with Nginx distributing traffic to backend Apps communicating natively with Redis/PostgreSQL. |
| **Enterprise Observability** | Configured `rsyslog` to pipe deep `vrrpd/ospfd` transition logs into **Grafana + Loki** for visual cluster debugging. |

## 🔥 Technical Differentiators

What distinguishes this project is the **fidelity of the simulation**:
1. **Network Hardware Equivalency:** R1, R2, and R3 are not ordinary linux containers. They run complex daemons (`zebra`, `ospfd`, `vrrpd`, `charon`) that modify low-level Linux Kernel network logic (FIBs).
2. **True Packet-level Access Control:** NGINX handles HTTP (L7), but we integrated `iptables` rules mapping `-A PREROUTING` and `-A FORWARD` directly on routers. 
3. **Observability Beyond Application:** Our Grafana stack doesn't just read JSON HTTP logs. Native C-binary Syslogs generated by the Daemons are ingested by Promtail, allowing Network Administrators to graph OSPF flap occurrences immediately.

## 🔐 Proof of Execution (IPsec Validation)

A critical component of this topology was the Site-to-Site VPN.

```console
# docker exec R1 ipsec status
Security Associations (2 up, 0 connecting):
      branch[2]: ESTABLISHED 50 minutes ago, 172.30.1.1[172.30.1.1]...172.30.1.2[172.30.1.2]
      branch{3}:  INSTALLED, TUNNEL, reqid 1, ESP SPIs: cc6a1664_i c83e457a_o
      branch{3}:   172.20.10.0/24 172.20.30.0/24 === 172.20.20.0/24
```
**Observation:** The Security Association (SA) successfully masks three vital subnets underneath a hardware tunnel (`ESP` payload). Traffic generated from the Branch can interact with `LAN A` completely obscured from the simulated ISP layer.
